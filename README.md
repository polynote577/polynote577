<summary><b>🔹 Дефект №1 | Рассинхронизация статуса документа при поисковом запросе через API (GET /v1/documents/search) после успешного подписания (POST /signdoc)</b></summary>

### **ID:** #JIRA-4092

**Severity**: Средний

**Priority**: Высокий

**1. Описание:**
При успешном выполнении запроса на подписание документа (POST /signdoc) статус документа в БД изменяется на SIGNED, однако поисковый запрос GET /v1/documents/search продолжает возвращать статус PENDING. В логах в момент воспроизведения наблюдаются ошибки подключения к Redis (timeout), которые могут быть связаны с отсутствием обновления поискового кэша.

**2. Предварительные шаги:**
1. Пользователь авторизован в системе
2. Получен валидный токен доступа
3. В системе существует документ со статусом DRAFT
4. Идентификатор документа имеет вид f001-4826-a813-9876543210ab

**3. Шаги воспроизведения:**

1. Отправить запрос на подписание документа:
> POST /v1/documents/f001-4826-a813-9876543210ab/signdoc
2. Убедиться, что запрос выполнен со статусом 200OK.
3 Выполнить SQL-запрос:
> SELECT status
> FROM document_registry
> WHERE id = 'de305d54-f001-4826-a813-9876543210ab';
4. Выполнить поисковый запрос:
> GET /v1/documents/search?query=f001-4826-a813-9876543210ab
5. Сравнить значение поля status в ответе API со значением в базе данных

**4. Фактический результат:**
1. Запрос на подписание выполняется со статусом 200OK
2. В базе данных статус документа изменяется на SIGNED
3. Поисковый запрос GET /v1/documents/search возвращает статус документа PENDING, который не соответствует текущему значению в БД

**5. Ожидаемый результат:**
1. Запрос на подписание выполняется со статусом 200OK
2. В базе данных статус документа изменяется на SIGNED
3. Поисковый запрос GET /v1/documents/search возвращает статус документа SIGNED

**6. Прикрепленные медиафайлы:**
- Скриншот с фактическим результатом поискового запроса GET
- Скриншот фактического ответа поискового запроса GET в Postman/Bruno
- Скриншот из БД с текущим статусом документа
- Скриншот/Ссылка на логи из Опеншифта с таймаутом подключения к Redis
  
**7. Дополнительная информация:**
Во время воспроизведения в логах сервиса обновления кэша наблюдаются ошибки подключения к Redis, например:
RedisTimeoutException: Unable to connect to Redis. Ошибки могут быть связаны с тем, что после изменения статуса документа не произошло обновление поискового кэша.

<summary><b>🔹 Дефект №2 | При переходе по прямой ссылке с document_id в фигурных скобках авторизованный пользователь не может скачать документ</b></summary>

### **ID:** #JIRA-4093

**Severity**: Средний

**Priority**: Высокий

**1. Описание:**
После перехода по прямой ссылке на скачивание файла, содержащей значение document_id в фигурных скобках ({}), скачивание файла не выполняется. Несмотря на наличие активной пользовательской сессии выполняется штатное автоматическое перенаправление через шлюз авторизации. После возврата параметры запроса теряются, вследствие чего скачивание документа не выполняется.

**2. Предварительные шаги:**
1. В системе существует документ, доступный для скачивания
2. Получен идентификатор документа (document_id)
3. Пользователь авторизован в системе (активна пользовательская сессия)
4. Сформирована прямая ссылка на скачивание файла с использованием значения document_id в фигурных скобках

**3. Шаги воспроизведения:**
1. Открыть браузер
2. Убедиться, что пользователь уже авторизован в системе
3. Вставить в адресную строку браузера ссылку вида:
>https://storage.internal/download?document_id={act-9912-prod}&type=storage

где document_id - id предзагруженного документа, type - наименование хранилища

4. Нажать Enter

**4. Фактический результат:**
1. Скачивание файла не выполняется
2. Выполняется автоматическое перенаправление через шлюз авторизации
3. После завершения перенаправления открывается пустая страница
4. В итоговом URL отсутствуют параметры document_id и type

**5. Ожидаемый результат:**
При переходе по прямой ссылке пользователь с активной сессией должен автоматически получить файл без повторной авторизации. Все параметры запроса должны сохраняться при перенаправлении, а скачивание должно завершаться успешно (HTTP 200)

**6. Прикрепленные материалы:**
- Скринкаст воспроизведения проблемы
- Скриншот вкладки Network с цепочкой перенаправлений
- Скриншот адресной строки до и после перенаправления (если хорошо демонстрирует потерю параметров)

**7. Дополнительная информация:**
В ходе анализа цепочки запросов установлено, что после автоматического перенаправления через шлюз авторизации из URL исчезают параметры document_id и type, вследствие чего сервис скачивания не получает необходимые параметры запроса.

### ✔️ Чек-листы

<details>
<summary><b>🔹 Чек-лист | Интеграционная проверка фильтрации ключей шифрования</b></summary>
Данный чек-лист предназначен для экспресс-проверки механизма фильтрации ключей шифрования по идентификатору компании (CID) при инициализации Формы подписания документов.

### 1. Подготовка окружения 
- [ ] Ключи установлены локально: **А** (CID `00xxxxxxxxxx`), **Б** (CID `99xxxxxxxxxx`), **В** (CID `00yyyyyyyyyy`), **Г** (без CID).
- [ ] Файлы для типов `Type_A`, `Type_B`, `Type_C`, `Type_D` и `Key_Storage` подготовлены.
- [ ] Опции просмотра, загрузки и подписания документов активны в панели администратора.

### 2. Тестирование логики фильтрации (Тип шифрования: Type_A)
- [ ] **Фильтрация отключена:** без передачи CID доступны все ключи (**А, Б, В, Г**).
- [ ] **Валидный CID с ведущими нулями:** при вводе CID Ключа А (`00xxxxxxxxxx`) доступен только **Ключ А**.
- [ ] **CID без ведущих нулей:** при вводе CID Ключа Б (`99xxxxxxxxxx`) фильтрация игнорируется, доступны все ключи.
- [ ] **Несуществующий CID:** при вводе отсутствующего в системе CID список ключей пуст, выводится предупреждение.
- [ ] **Невалидный формат CID:** при вводе некорректного значения (например, `123456789A`) фильтрация игнорируется.

### 3. Кросс-функциональные проверки (Другие типы шифрования)
- [ ] **Type_B:** Проверить, что при вводе CID Ключа А (`00xxxxxxxxxx`) список фильтруется до одного Ключа А, подписание проходит успешно
- [ ] **Type_C:** Проверить, что при вводе CID Ключа А (`00xxxxxxxxxx`) список фильтруется до одного Ключа А, подписание проходит успешно
- [ ] **Type_D:** Проверить, что при вводе CID Ключа А (`00xxxxxxxxxx`) список фильтруется до одного Ключа А, подписание проходит успешно

### 4. Интеграция с внешним облачным хранилищем (Key_Storage)
- [ ] **Ввод CID:** Чекбокс активен, введен CID `00xxxxxxxxxx`, загружен файл под сценарий `Key_Storage`
- [ ] **Облачная авторизация:** Фильтрация локальных ключей по CID не применяется к облачному хранилищу
- [ ] **Подписание:** Окно авторизации в облачном сервисе открывается без ошибок, подписание по токену завершено успешно (Статус-код `200 OK`)
</details>
